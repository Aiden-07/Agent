# 版本管理机制审查报告与重构设计文档

## 1. 当前实现逻辑与业务规则的差异分析

### 1.1 快照创建流程 (规则1)
- **要求**：完整捕获页面功能、配置变更、数据更新等内容。
- **现状**：目前 `VersionManager.captureState()` 仅仅捕获了左侧导航栏菜单项的显示/隐藏状态（存储在 `features` 数组中）。
- **差异**：**严重不足**。当前的快照系统只是一个“菜单开关模拟器”，并没有真正保存主内容区域的任何状态（如策略配置表单的值、列表数据、DOM 结构的动态修改等）。

### 1.2 快照隔离机制 (规则2)
- **要求**：新增或修改的内容不得与历史快照产生关联，数据完整且独立。
- **现状**：因为这是一个 SPA（单页应用），所有的 DOM 都是共享内存的。当用户在“当前版本”删除了一个 `div` 时，这个操作直接修改了浏览器的 DOM 树。由于快照没有保存 DOM 或数据的深拷贝，切换版本只是简单地切换了菜单的可见性，被删除的 `div` 依然处于被删除的状态。
- **差异**：**完全失效**。历史快照被当前的临时 DOM 修改“污染”了，因为系统根本没有隔离机制。之前添加的 `window.loadView(viewName)` 强制刷新只是一个治标不治本的补丁，它只能从磁盘重新加载原始 HTML，而无法恢复到“创建该版本快照时”的确切状态。

### 1.3 时间点还原能力 (规则3)
- **要求**：准确还原保存时刻的完整功能状态和数据内容（布局、配置、数据）。
- **现状**：目前只能还原导航菜单的可见性，以及通过重新加载原始 HTML 文件来粗暴地重置页面。
- **差异**：**无法实现真正的时间点还原**。系统无法还原用户在某个特定版本中填写的表单数据、动态添加的元素或修改的全局变量。

## 3. 更新后的版本管理流程设计文档

为了满足上述三大严格的业务规则，我们需要将版本管理从“轻量级的菜单切换”升级为**“重量级的状态与DOM快照管理”**。

### 3.1 数据结构设计 (Snapshot Model)
每个版本的数据结构必须包含：
```json
{
  "id": "v-123456789",
  "version": "1.3.0",
  "remarks": "用户自定义的完整快照",
  "timestamp": 123456789000,
  "snapshot": {
    "navState": ["nav-agent", "nav-knowledge"], // 导航状态
    "globalData": {
       "mockDocs": [...], // 拦截并保存核心业务数据
       "knowledgeData": [...]
    },
    "domSnapshots": {
       "knowledge-strategy-config": "<html>...</html>", // 特定关键页面的完整 HTML 字符串深拷贝
       "knowledge-list": "<html>...</html>"
    }
  }
}
```

### 3.2 流程设计

1.  **保存快照 (Create)**：
    *   遍历导航栏获取激活状态。
    *   序列化核心全局变量（使用 `JSON.stringify` 进行深拷贝）以确保数据隔离。
    *   如果需要页面级别的精准还原，抓取当前 `#main-content-area` 内的 `innerHTML`。
    *   将这些数据打包存入 LocalStorage。

2.  **切换快照 (Restore)**：
    *   清空当前 DOM。
    *   从 LocalStorage 读取目标快照。
    *   反序列化全局变量，覆盖当前的内存变量（如 `window.mockDocs = JSON.parse(snapshot.mockDocs)`）。
    *   如果存在 DOM 快照，直接将 `innerHTML` 注入对应的容器，并重新绑定事件委托。
    *   触发一个全局自定义事件（如 `vagent-version-restored`），通知各个业务模块重新渲染或刷新其内部状态。

### 3.3 隔离性保障
*   所有的读写操作必须基于深拷贝。
*   当前页面的任何修改，只要不触发“保存为新版本”或明确的 API 提交，就只是当前内存的脏数据。切换版本时，内存会被快照数据强制覆盖，从而保证历史快照不被污染。

## 4. 测试用例验证设计

这些测试用例可以通过自动化测试脚本（如 Playwright/Cypress）或手动执行来验证。

### 测试用例 1：快照隔离性测试 (Isolation)
*   **前置条件**：系统处于 `v1.0.0`，包含数据 A。
*   **操作**：
    1.  用户在界面上手动删除数据 A，此时界面显示数据 A 已消失。
    2.  用户不保存新快照，直接在右侧抽屉中点击切换到 `v1.1.0`。
    3.  用户再次点击切换回 `v1.0.0`。
*   **预期结果**：数据 A 重新出现在界面上。这证明了临时修改（删除 A）没有污染 `v1.0.0` 的快照数据。

### 测试用例 2：数据完整性捕获测试 (Integrity)
*   **前置条件**：系统处于任意版本。
*   **操作**：
    1.  在“策略配置”页面修改表头配置为“第 3 行”。
    2.  点击“新增版本快照”，保存为 `v2.0.0`。
    3.  切换到 `v1.0.0`。
    4.  切换回 `v2.0.0`。
*   **预期结果**：“策略配置”页面的表头配置应准确显示为“第 3 行”，而不是默认的“第 1 行”。

### 测试用例 3：时间点还原准确性测试 (Restoration)
*   **前置条件**：准备两个连续的快照 `v1` (含功能模块 X) 和 `v2` (含功能模块 X, Y)。
*   **操作**：
    1.  在 `v2` 下，进入功能模块 Y 并填写表单数据。
    2.  切换到 `v1`。
*   **预期结果**：不仅表单数据消失，整个功能模块 Y 在左侧导航和路由中都应该不可见且不可访问，系统严格表现为创建 `v1` 时的状态。
